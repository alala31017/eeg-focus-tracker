
# Personal Focus Tracker with Muse S Athena
 
[English README is available here](./README.en.md)
 
Muse S Athenaヘッドセットからリアルタイムで脳波(EEG)データを取得し、学習済みのLightGBMモデルで集中状態を判定して、日英バイリンガル対応のStreamlitダッシュボードに表示する個人開発プロジェクトです。事前に記録したセッションを再生する「デモモード」も搭載しており、実機を持っていなくてもアプリの動作を確認できます。
 
**公開デモ:** [https://muse-eeg-focus-tracker.streamlit.app/](#)

## 概要
 
このプロジェクトは、**データ収集→前処理→モデル学習→リアルタイム推論アプリの構築**という一連の流れを、企画から実装まで一貫してPython環境で行ったものです。EEGの生データを録音するスクリプトの作成から、信号処理パイプラインの実装、複数モデルの比較検討、最終的なStreamlitアプリのデプロイまで、機械学習プロジェクトの全工程を経験することを目的としています。

コンシューマー向けBLE脳波測定ヘッドセットから取得した生のEEGデータを、信号処理と機械学習のパイプラインに通し、リアルタイムで「集中度」の確率スコアを算出します。スコアは3秒ごとに更新され、脳波バンドパワー、チャンネル活動、心拍数、セッション履歴とともに表示されます。
 
システム設計：
 
- **データ収集** — ベースライン・集中タスク・気が散るタスクを組み合わせた録音セッションをバイナリファイルとして保存
- **オフライン前処理・学習** — 信号フィルタリング、エポック分割、特徴量抽出、LightGBM分類器の学習。`notebooks/`配下にステップごとに記録
- **リアルタイム推論** — BLEストリーム → 信号処理パイプライン → モデル推論 → Streamlit UI
- **デモモード** — 記録済みの`.bin`セッションを同じ推論パイプラインで再生する仕組み。Bluetoothデバイスが無くてもアプリを試せる（公開デプロイ版で使用）
---
 
## ハードウェア
 
- **デバイス:** Muse S Athena
- **チャンネル:** TP9, AF7, AF8, TP10（4電極）
- **サンプリングレート:** 256 Hz
- **OS:** ライブモードはmacOSのみ対応（BLEスタックは`bleak` / CoreBluetoothを使用）。デモモードはOSの制限なし

## インストール
 
Python 3.10.1が必要です。
 
```bash
git clone https://github.com/your-username/eeg-focus-tracker.git
cd eeg-focus-tracker
 
python3.10 -m venv .venv
source .venv/bin/activate
```
 
### デモモードのみ（Bluetoothデバイス不要）
```bash
pip install -r requirements.txt
```
 
### ライブモードを含むフルセットアップ（Muse S Athenaヘッドセットが必要）
```bash
pip install -r requirements.txt -r requirements-live.txt
```
 
**macOSのBluetooth権限について:** 初回実行時、macOSがBluetoothへのアクセス許可を求めます。システム設定 → プライバシーとセキュリティ → Bluetoothで許可してください。デバイスが見つからない場合は、Bluetoothを一度オフ・オンにして再試行してください。

## 使い方

> アプリを試すだけであれば、1〜3は読み飛ばして[4. リアルタイム推論](#4-リアルタイム推論)に進んでください。`models/`には学習済みのモデルがすでに含まれているため、データ収集や再学習を行わなくてもデモモードはそのまま動作します。
 
### 1. データ収集
 
30分のレコーディングセッション（ベースライン → タスク1 → 休憩 → タスク2）を実行します：
 
```bash
python -m src.eeg_focus_tracker.data_collection.data_recording
```
 
セッション番号の奇偶によってfocus-first/distraction-firstの順序が切り替わります（奇数=focus先行、偶数=distraction先行）。
レコーディング完了後、`data/recordings/`に3種類のファイルが保存されます：生バイナリデータ（命名規則: `session{n}_{condition}_{timestamp}.bin`）、各タスクブロックの開始・終了時刻を記録したイベントCSV（前処理時の参照用）、セッション概要を記録したメタデータJSONです。
 
### 2. 前処理
 
1セッション分のバイナリデータを読み込み、信号フィルタリング・アーティファクト除去・エポック分割・特徴量抽出を行い、26列（25特徴量＋ラベル）のCSVとして保存します。
 
```bash
python -m src.eeg_focus_tracker.data_collection.preprocessing
```
 
### 3. 学習
 
`03_modeling.ipynb`で選定したハイパーパラメータを用いて、ノイズを除いたクリーンなセッションデータを用いて最終的なLightGBM分類器を学習します（LightGBM + Leave-One-Session-Out CV。KFoldよりLOSOを採用した理由は[モデル](#モデル)セクションを参照）。
 
```bash
python -m src.eeg_focus_tracker.training.model
```
 
`focus_model.pkl`、`scaler.pkl`、`feature_names.pkl`が`models/`に保存されます。
 
### 4a. リアルタイム推論 — ライブモード
 
アプリを起動します：
 
```bash
streamlit run app/streamlit-app.py
```
 
ランディング画面で**ライブモード**を選択し、**ライブセッションを開始**をクリックします。これにより`realtime_pipeline.py`がバックグラウンドのサブプロセスとして起動し、Museデバイスをスキャン・BLE接続・ストリーミングを開始します。接続後、30秒のベースライン計測が始まり、完了後はダッシュボードが約3秒ごとに集中スコア・バンドパワー・チャンネル活動・心拍数を更新します。
 
### 4b. リアルタイム推論 — デモモード
 
ランディング画面で**デモモード**を選択します。自分で録音した`.bin`セッションをアップロードするか、**サンプルセッションを使用**で同梱のサンプル録音を再生できます。再生されたデータはライブモードと**同じ推論パイプライン**（`process_eeg`）で処理されるため、デモモードはヘッドセット無しでもライブ体験を再現します。再生速度は4倍速です。
 
### セッションの終了
 
**セッション終了**（ライブモード）をクリック、またはデモの再生終了を待つと、サマリー画面が表示されます：セッション時間、平均/最高/最低集中度、色分けされた集中タイムライン、各集中ゾーンの滞在時間割合、平均バンドパワー、自動生成されたインサイトのリストです。新しいセッションの開始、同じデモファイルのリピート再生、または **← モードを変更**によるモードの切り替えが可能です。

## 信号処理パイプライン
 
入力されたEEGウィンドウは以下の手順で処理されます：
 
1. **バンドパスフィルタ** — 1〜30 Hz（バターワース4次）
2. **エポック分割** — 6秒ウィンドウ・3秒ストライド（50%オーバーラップ）
3. **アーティファクト除去** — サンプルの20%以上が±380µVを超えるウィンドウは除外
4. **FFT / Welch法によるPSD推定** — θ波（4〜8Hz）、α波（8〜13Hz）、β波（13〜30Hz）のバンドパワー抽出
5. **特徴量エンジニアリング** — 4チャンネル×各種指標で計25特徴量：
   - チャンネル別: theta, alpha, beta, theta/alpha比, beta/alpha比, engagement index
   - 全体: frontal alpha asymmetry（AF7 − AF8）
6. **ベースライン正規化** — 各特徴量をセッション固有のベースライン中央値で除算（`value / baseline`）し、セッション間のばらつきを軽減

## ノートブック
 
`notebooks/`配下には、データ収集後の前処理からモデル選定までの過程を、3つのノートブックに分けて記録しています。
 
### `01_preprocessing.ipynb` — 前処理
 
1セッション分のデータを例に、EEG前処理パイプラインを1ステップずつ解説しています。バンドパスフィルタ、エポック分割とアーティファクト除去、FFTによるバンドパワー抽出、ベースライン正規化までを可視化しながら確認し、最後に同じパイプラインを全15セッションに適用して、学習用データセットを構築します。
 
### `02_exploratory.ipynb` — 探索的データ分析（EDA）
 
今プロジェクトで収集されたデータにおけるEEGバンドパワー特徴量が、集中状態と散漫状態の間で異なるかどうかを統計検定を用いて検証しています。信号品質指標に基づきノイズの多い3セッションを除外した「クリーン」データセットも作成し、全データセットとの比較を行っています。Shapiro-Wilk検定で正規性を確認した上で対応のあるt検定を採用し、多重比較に対するFDR補正も適用しています。
 
### `03_modeling.ipynb` — モデリング
 
Random Forest、LightGBM、SVMの3モデルを、K-Fold CVとLeave-One-Session-Out（LOSO）CVの両方で比較しています。LOSOを主要な評価指標として採用し、クリーンデータセットを用いたLightGBMを最終モデルとして選定しています。

## モデル
 
**分類器:** LightGBM。ラベルは2値（1 = 集中状態、0 = 散漫状態）。
 
### KFold vs LOSO（Leave-One-Session-Out）
 
最初の評価では標準的なKFold交差検証を使用し、高い精度が出ました。しかしKFoldはウィンドウ単位でデータを分割するため、同一セッション内のEEGウィンドウが持つセッション固有の特性（電極の接触状態、その日の精神状態など）が、たまたま同じフォールド内の学習用・テスト用に混在してしまい、見かけのスコアが過大評価されやすいという欠点があります。
 
実際の運用シナリオ（毎回新しいセッションでヘッドセットを装着する）に即した本来の汎化性能を測るため、**Leave-One-Session-Out（LOSO）交差検証**で再評価しました（1セッションずつを丸ごとホールドアウト）：
 
| モデル | KFold ROC-AUC | LOSO ROC-AUC | 差 |
|-------|----------------|--------------|-----|
| SVM (全セッション)       | 0.809 | 0.568 | 0.241 |
| LightGBM (クリーンセッション) | 0.759 | 0.587 | 0.172 |
| LightGBM (全セッション)  | 0.700 | 0.561 | 0.139 |
| RF (全セッション)        | 0.644 | 0.530 | 0.114 |
 
未知のセッションへの汎化性能を反映するため、**本プロジェクトではLOSOを主要な評価指標として採用しました。** 
 
### 最終モデルの性能（LightGBM、LOSO、クリーンセッション）
 
| 指標 | LOSOスコア |
|--------|-----------|
| ROC-AUC | 0.587 |
| F1 | 0.524 |
| Accuracy | 0.570 |
 
セッション4・7・8は、事前に定義した信号品質のしきい値（平均CV > 0.6、outlier率 > 15%、ラベルバランス比 < 0.7、平均θ/α比 > 1.6）に基づき「クリーン」なデータセットから除外しました（詳細な基準とセッション別の数値は`02_exploratory.ipynb`を参照）。これらのセッションを除外することで、全データセットに対してLOSO AUCが若干（約+0.03）改善しました。特徴選択（feature selection）も試しましたが、25特徴量全部を使う場合よりLOSO性能が改善することはありませんでした。
 
このモデルは個人専用です。単一の被験者で学習・調整されており、別の人に対する汎化は想定していません（再学習が必要です）。

## アーキテクチャに関する補足
 
### プロセス間通信
 
ライブモードの推論パイプライン（`realtime_pipeline.py`）は、Streamlit UIとは別のサブプロセスとして動作します。ローカル・単一ユーザー環境でのシンプルさを優先し、両プロセス間のデータ共有は`shared_state.py`（atomic write（`os.replace`）を用いたJSONファイルの薄いラッパー）を介して行っています。
 
以下のような制約が考えられるため、本格的な運用版であれば、Redisのpub/subやWebSocket接続に置き換え、複数セッションを安全に並行処理できるようにする必要があります。
- 複数ユーザーの同時アクセスには対応していません（公開デプロイのデモモードは、全アクセスユーザーで1つの状態ファイルを共有しています）。 同時にデモを実行する複数のユーザーアクセスがある場合、進捗が干渉する可能性があります。
- ファイルI/Oの遅延があるため高頻度の更新には不向きです。
